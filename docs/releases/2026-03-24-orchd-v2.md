# orchd v2

orchd v2 turns the original orchestration loop into a real project finisher.

The headline change is simple:

- initialize a repo
- describe the goal
- run `orchd finish`

In practice, many users will do this through an external coding agent. v2 now has a better handoff path for that style of usage too:

```bash
orchd agent-prompt orchestrator "your goal here"
```

Paste that into your agent and let it drive orchd for you.

From there, orchd can keep driving task spawn/check/merge/recovery/ideation until the work is actually blocked or complete.

## Release Highlights

### 1. `orchd finish`

New deterministic finisher mode that combines:

- adaptive verification
- aggressive recovery
- split-aware replanning
- reviewer-assisted risky merges
- follow-on ideation until `PROJECT_COMPLETE`

This is now the recommended default path for most users.

### 2. Better recovery

orchd now classifies failures and chooses a next action instead of treating every failure the same.

Examples:

- retry the same runner
- retry an alternate runner
- split and replan oversized work
- stop and mark `needs_input` when human input is genuinely required

### 3. Risk-aware merging

Merge policy is no longer one-size-fits-all.

- low-risk tasks can pass with lighter verification
- high-risk tasks can require broader validation
- approved review can satisfy risky merge policy when verification is good but not full

### 4. Swarm telemetry

`orchd state --json` now exposes richer machine-readable state:

- scheduler decisions and reasons
- finisher state
- orchestrator route/reminder metadata
- per-task routing, verification, recovery, review, merge-gate, and split metadata

This also powers better TUI and fleet visibility.

### 5. TUI and fleet upgrades

The terminal UI now shows:

- routing and runner selection
- recovery and failure state
- review and merge-gate state
- finisher and scheduler summaries

Fleet and doctor output now surface the same high-value signals.

### 6. Better docs and rollout guidance

v2 adds clearer guidance for:

- `orchd finish`
- swarm policy settings
- rollout from legacy behavior toward full v2 usage
- reviewer-aware merge flow

## Recommended User Path

For most users, v2 should feel like this:

```bash
orchd init .
orchd doctor
orchd plan "your goal here"
orchd finish
```

Advanced commands still exist, but they are no longer required to get real value from orchd.

If the human is operating through an external agent instead of the shell directly, the recommended entrypoint is:

```bash
orchd agent-prompt orchestrator "your goal here"
```
